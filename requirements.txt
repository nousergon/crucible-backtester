# Bumped 0.26.1 -> 0.28.5 for numpy 2 compat (config#2815) — NOT a drive-by
# bump, it's load-bearing: cvxpy>=1.9.2 below hard-requires numpy>=2.0.0
# (every cvxpy release from 1.8.0 on does), and 0.26.1 is fundamentally
# broken under numpy>=2 (`vectorbt/generic/enums.py` does `np.float_`,
# removed outright in numpy 2.0 -> AttributeError at `import vectorbt`,
# breaking vectorbt_bridge.py + every test importing it). The obvious next
# patch, 0.26.2, only band-aids ONE of several numpy-2 breaks (it wraps
# `numpy.lib.stride_tricks._broadcast_shape`, a removed private symbol, in a
# try/except) and still declares `numpy<2.0.0` in its own wheel metadata —
# pip correctly refuses to resolve it against numpy 2, and forcing the
# combo anyway still hits the `np.float_` AttributeError above at import
# time. 0.27.x+ removed all removed-numpy-alias usage; 0.27.3 specifically
# was ruled out here because it AttributeErrors against pandas~=2.2's
# `DatetimeIndex.freq` (unrelated pandas-side incompatibility, not numpy).
# 0.28.5 has neither problem: clean `import vectorbt` under numpy 2.4.6,
# functional Portfolio.from_signals()/.sharpe_ratio()/.sortino_ratio()
# smoke-tested directly, and the full backtester suite passes at 0.28.5
# (3103/3105; the other 2 failures are a pre-existing RAM-headroom guard
# unrelated to this change — see PR). No API-signature diff found between
# 0.26.x/0.27.x/0.28.x for the Portfolio methods vectorbt_bridge.py calls.
